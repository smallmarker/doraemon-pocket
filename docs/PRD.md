# Doraemon Pocket 产品需求文档（PRD）

> 版本：v0.1  
> 状态：首版定义  
> 产品形态：纯前端、静态部署的开源工具导航搜索站

## 1. 产品概述

**Doraemon Pocket** 是一个以搜索为唯一核心入口的在线工具聚合网站。

用户无需浏览复杂分类或记忆工具网址，只需输入关键词，例如“时间戳”“JSON 格式化”“图片压缩”，网站便从预先维护的工具库中筛选并展示相关工具的简介与直达链接。

产品不执行用户任务，不生成内容，也不依赖 AI 或后端服务；它的职责是帮助用户快速找到合适的第三方工具。

## 2. 背景与问题

现有工具导航站通常将大量网站按分类铺陈，信息密度高、选择成本高。用户常常只知道自己想做什么，却不记得合适工具的名称或网址。

本项目希望用一个“口袋”式搜索入口解决这个问题：需要什么工具，就搜索什么工具。

## 3. 产品目标

### 3.1 首版目标

- 提供一个极简首页，视觉重点仅为搜索框。
- 支持中文关键词、工具名称、标签和别名的本地模糊搜索。
- 在结果页展示少量高相关工具，并可一键打开外部链接。
- 使用静态数据文件保存工具资料，可部署至 GitHub Pages。
- 让开源贡献者可通过修改数据文件补充、修复工具。

### 3.2 非目标

- 不提供登录、账号体系或跨设备收藏同步。
- 不调用大模型，不进行对话式问答或任务执行。
- 不抓取、镜像或托管第三方工具内容。
- 不建设后台管理系统、数据库或服务端 API。
- 首版不追求收录所有工具，也不展示复杂分类导航。

## 4. 目标用户

| 用户 | 典型需求 | 核心诉求 |
| --- | --- | --- |
| 开发者 | JSON 格式化、时间戳转换、正则测试 | 少记网址，快速直达 |
| 日常办公用户 | PDF 合并、图片压缩、二维码生成 | 找到免费、易用的工具 |
| 内容创作者 | 图片去背景、配色、图标、字幕 | 快速发现合适的轻量工具 |

## 5. 核心用户流程

```text
访问首页
  → 在口袋搜索框输入需求/关键词
  → 前端从本地工具库进行匹配与排序
  → 展示相关工具结果
  → 用户点击“打开工具”跳转至第三方网站
```

## 6. 功能需求

### 6.1 首页搜索

- 首页只承担“开始搜索”这一件事，不展示工具分类墙、推荐列表或复杂导航。
- 首屏由**哆啦A梦人物视觉**与**口袋式搜索框**共同构成唯一视觉主角：人物位于搜索框一侧或后方，搜索框在视觉上与人物腹部口袋连接，形成“从口袋里取工具”的直觉。
- 人物视觉应作为独立资源/组件管理，保证后续可替换不同姿势、主题或授权素材，而不影响搜索功能和页面布局。
- 搜索框占据视觉中心，桌面端自动获得焦点；移动端保证人物不遮挡输入与提交操作。
- 项目名称、简短文案可置于搜索框上方或下方，但信息层级必须弱于人物与搜索框。
- Placeholder 示例：`想找什么工具？例如：时间戳、JSON 格式化、图片压缩`
- 支持 Enter 提交搜索。
- 空搜索不跳转或提示用户输入关键词。

#### 首页布局要求

| 区域 | 内容 | 设计要求 |
| --- | --- | --- |
| 顶部 | 轻量 Logo / 项目名称 | 可选；不得抢占首屏注意力 |
| 首屏主视觉 | 哆啦A梦人物 | 体现友好、取道具的动作或姿态；不得干扰搜索可读性 |
| 搜索核心区 | 搜索输入框、搜索按钮/图标 | 具备“口袋开口”联想；输入高度和点击区域适合移动端 |
| 辅助文案 | 一句产品说明、搜索示例 | 只做引导，不显示分类和工具列表 |
| 页脚 | GitHub、贡献入口、免责声明 | 低干扰，放在首屏边缘或下方 |

#### 首页交互状态

- 默认：显示人物主视觉、空搜索框及示例提示。
- 聚焦：搜索框边框/阴影轻微强化；可显示 3–5 个示例关键词，不展示全量分类。
- 输入中：可显示清除按钮；按 Enter 或点击搜索按钮进入结果页。
- 移动端：人物可缩小或移动至搜索框上方，搜索框保持在首屏可见区域。

### 6.2 搜索匹配

- 搜索范围：`name`、`description`、`tags`、`aliases`、`scenarios`。
- 支持大小写不敏感的英文搜索。
- 支持中文关键词与常见别名，例如“json美化”匹配“JSON Formatter”。
- 允许模糊匹配，用于处理轻微拼写差异或不完整输入。
- 结果按相关度、人工优先级排序；首版最多默认展示 8 条。

### 6.3 搜索结果

结果页使用**统一结果卡片模板**。新增工具只需补充数据，不应需要为某个工具单独设计或编写页面代码。

结果页结构：

```text
顶部搜索栏（可再次搜索）
  → 当前搜索词与结果数量
  → 统一工具结果卡片列表
  → 无结果状态 / GitHub 贡献入口
```

每条结果卡片固定包含：

- 工具名称
- 一句话说明
- 标签（最多 3 个）
- “打开工具”按钮
- 外部链接标识

可选展示字段（数据存在时显示）：

- 免费 / 需登录等轻量状态标签
- 使用场景短句
- 工具图标或首字母占位图

#### 统一结果卡片模板

| 区域 | 数据来源 | 规则 |
| --- | --- | --- |
| 工具识别区 | `icon`（可选）、`name` | 左侧小图标/占位，名称为第一视觉层级 |
| 描述区 | `description` | 最多两行，超出省略，避免卡片高度失控 |
| 标签区 | `tags`、`free`、`requiresLogin` | 主题标签最多 3 个；状态标签统一样式 |
| 场景区 | `scenarios`（可选） | 最多展示 1 条，用于补充“何时使用” |
| 行动区 | `url` | 固定文案“打开工具”，在新标签页跳转 |

卡片布局原则：

- 桌面端采用单列、宽内容区，避免工具图标瀑布流造成信息噪音。
- 卡片的标题、说明、标签、按钮位置在所有结果中保持一致。
- 点击整张卡片或“打开工具”均可跳转；按钮仍需保留以明确操作意图。
- 结果默认展示 8 条；超出时通过“显示更多”逐步加载，避免一次展示过多。
- 移动端卡片纵向排列，按钮占满或接近占满可点击宽度。

结果为空时显示友好空状态：

> 口袋里暂时没有找到这个工具。欢迎到 GitHub 提交一个工具推荐。

### 6.4 外链行为

- 外部工具在新标签页打开。
- 使用 `noopener noreferrer`，避免外链获得原页面控制权。
- 数据中可标注工具是否免费、是否需要登录；首版仅作为信息展示，不做筛选器。

### 6.5 数据维护与开源贡献

- 工具资料存放在可读、可编辑的静态 JSON 文件中。
- README 提供新增工具、修改别名、报告失效链接的说明。
- 每个工具须至少包含名称、有效 URL、简介、一个标签或别名。

## 7. 工具数据模型

建议数据文件位置：`src/data/tools.json`。

```json
{
  "id": "json-formatter",
  "name": "JSON Formatter",
  "url": "https://example.com",
  "description": "在线格式化、校验和压缩 JSON。",
  "tags": ["开发", "JSON", "格式化"],
  "aliases": ["json美化", "json校验", "json压缩", "接口数据"],
  "scenarios": ["检查接口返回数据", "整理 JSON 文本"],
  "free": true,
  "requiresLogin": false,
  "priority": 100
}
```

字段约束：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `id` | 是 | 唯一、URL 安全的小写标识 |
| `name` | 是 | 工具显示名称 |
| `url` | 是 | 完整 HTTPS 地址 |
| `description` | 是 | 不超过约 50 个中文字符的用途说明 |
| `tags` | 是 | 1–5 个主题标签 |
| `aliases` | 推荐 | 用户可能输入的同义词、俗称或英文名 |
| `scenarios` | 可选 | 真实任务或使用情境 |
| `free` | 可选 | 是否具备免费可用版本 |
| `requiresLogin` | 可选 | 是否通常需要登录 |
| `priority` | 可选 | 同等相关度下的人工排序权重 |

## 8. 首版工具范围

首版目标：收录约 50–100 个经人工检查的工具，优先覆盖高频需求。

- 开发：JSON、正则、时间戳、编码转换、SQL、API 调试
- 文件：PDF、图片、音视频、压缩、格式转换
- 效率：二维码、文本处理、Markdown、OCR、临时邮箱
- 设计：配色、图标、字体、截图、图片处理

收录原则：链接有效、用途清晰、体验可接受；不以数量为目标。

## 9. 技术与部署约束

- 采用纯前端静态架构。
- 工具数据随构建产物发布到浏览器端。
- 搜索在浏览器本地执行；可使用 Fuse.js 或等价轻量搜索方案。
- 不需要服务端、数据库、密钥或用户数据采集。
- 通过 GitHub Pages 部署；推送主分支后由 GitHub Actions 自动构建发布。
- 优先适配桌面和移动端；页面需具备基础无障碍支持（label、键盘操作、可读对比度）。

## 10. 视觉与内容原则

- 极简：首页不展示工具分类墙、排行榜或冗余内容。
- 角色与功能结合：首页的人物主视觉必须服务于“口袋搜索”的概念，而不是单独陈列的装饰插图。
- 聚焦：搜索框是首页唯一的主操作；结果页顶部搜索栏是唯一的主操作。
- 统一：所有工具均使用同一张结果卡片模板，不为工具类型引入不同的结果页版式。
- 克制：结果以单列列表或低干扰卡片呈现，不做图标瀑布流。
- 层级：工具名称 > 工具用途 > 标签/场景 > 外链动作；不要让标签或图标喧宾夺主。
- 响应式：首页在窄屏下优先保证搜索可用，结果页在窄屏下优先保证阅读和点击。
- 友好：文案采用“从口袋中取工具”的轻松语气；人物资源的来源、使用范围与替换方式应在项目中单独说明。

## 11. 首版验收标准

- [ ] 首页打开后可立即输入并搜索。
- [ ] 搜索“时间戳”“json美化”“图片压缩”等预置关键词时，能返回正确工具。
- [ ] 工具名称、标签、别名均能参与匹配。
- [ ] 搜索结果能在移动端正常阅读和点击。
- [ ] 点击外部链接会在新标签页打开。
- [ ] 无结果时显示空状态和贡献入口。
- [ ] 全站不依赖后端接口，构建产物可部署到 GitHub Pages。
- [ ] 新增一条工具数据不需要修改搜索业务代码。
